# voicechat-mac

Native SwiftUI voice + chat client for Qwen3-Omni-30B-A3B running locally on
Apple Silicon. Pairs the omlx OpenAI-compatible server with a Swift front-end
that does mic capture and audio playback directly via `AVAudioEngine` — no
browser, no Electron, sub-50ms input latency.

## Pipeline

```
  mic ──►  AVAudioEngine (16 kHz mono PCM16 WAV)
       ──► POST /v1/audio/transcriptions   (Whisper / Qwen3-ASR via omlx)
       ──► POST /v1/chat/completions       (Qwen3-Omni 8-bit + tool calls)
       ──► local tool execution            (Swift handlers, OpenAI tools spec)
       ──► POST /v1/chat/completions       (loop until no more tool_calls)
       ──► POST /v1/audio/speech           (Kokoro 82M TTS via omlx)
       ──► AVAudioPlayer                   (speaker)
```

The chat model uses **native `<tool_call>` function calling** built into
Qwen3-Omni — omlx parses it into OpenAI-style `tool_calls` automatically.

## Why a TTS sidecar instead of Qwen3-Omni's Talker

Qwen3-Omni's own Talker (24 kHz speech-out, voices Ethan/Chelsie/Aiden) is
**not yet wired up in the current `mlx-vlm` port** of the MLX checkpoint. Audio
*in* (understanding) works; audio *out* needs a separate model. Kokoro 82M is
the lowest-latency MLX TTS available and is already in your mlxmgr catalog as
`kokoro`. Swap it for `voxcpm2-8bit` or `omnivoice` in `OmlxConfig` if you want
voice cloning or higher quality.

## Prerequisites

1. **Models pulled into the HF cache** — from the project root:
   ```bash
   ./download.py qwen3-omni-30b-a3b-8bit   # ~38 GB (this script kicks it off)
   ./mlxmgr.py install kokoro              # ~0.3 GB
   ./mlxmgr.py install omnivoice           # optional, ~2 GB, for richer TTS
   ```
   For STT, install Whisper or Qwen3-ASR via omlx's own model manager (it
   pulls from `mlx-community` on first use).

2. **omlx server running** — Apache 2.0 menu-bar app + OpenAI-compatible
   server. Cloned into `../ext/omlx/`. Install into the project venv (the
   system Python on macOS rejects bare `pip install` per PEP 668), then
   start it. omlx auto-discovers any model already in `~/.cache/huggingface`
   (the default `--hf-cache` behavior), so no model arg is needed:
   ```bash
   ../.venv/bin/pip install -e "../ext/omlx[mcp]"
   ../.venv/bin/omlx serve --port 8000
   # or `../.venv/bin/omlx start` for the managed background daemon.
   ```
   Fish users can `source ../.venv/bin/activate.fish` once and then call
   `omlx` directly. See `../ext/omlx/README.md` for the full quickstart.

3. **Swift 5.9+ / Xcode CLT** — `swift --version` should print 5.9 or later.

## Build & run

```bash
./build.sh
open VoiceChat.app
```

The first launch prompts for microphone access (driven by
`Resources/Info.plist`'s `NSMicrophoneUsageDescription`). Grant it.

## Voice presets

The picker in the composer bar lists voices loaded from `voices/voices.json`.
Two flavors:

- **Kokoro** — fixed named voices (`af_heart`, `am_adam`, etc.). Deterministic
  by construction; ~0.3 GB model.
- **OmniVoice designed** — voices generated by sending rich text descriptions
  ("warm narrator", "gravelly detective") to OmniVoice's `instruct` channel.
  Deterministic after design because we save the captured WAV and replay it as
  `ref_audio` on every turn.

### Designing OmniVoice voices

With omlx running, generate a fresh manifest:

```bash
./design_voices.py
./build.sh             # rebundle so the .app picks them up
```

`design_voices.py` iterates the `DESIGNED` list at the top of the script
(edit it to add your own voices), pings `/v1/audio/speech` with the model set
to OmniVoice, captures the WAV, and writes `voices/voices.json` plus
`voices/<id>.wav`. Re-run with `--only <id1>,<id2>` to regenerate just a few.

If `voices/` is empty, the app falls back to a single built-in Kokoro voice
(`af_heart`) so you can use it without running the design step.

## Usage

- **Push-to-talk:** hold the spacebar (when the text field isn't focused),
  release to send. Or click the mic button.
- **Type to chat:** plain text in the composer.
- **Voice reply toggle:** turn off to get text-only replies (skips TTS).

## Tools shipped

| Tool | Purpose |
|---|---|
| `get_current_time` | ISO-8601 timestamp. |
| `list_directory` | `path` → entries in that directory. |
| `read_file` | `path` → first 8 KB of UTF-8 file contents. |
| `run_shell` | `command` → `/bin/sh -c <command>` output. **No sandbox.** |

Add more by implementing `ToolHandler` in `Sources/VoiceChat/ToolRegistry.swift`
and registering with `tools.register(...)` in `ChatSession.init`.

## Configuration

Edit `OmlxConfig` in `Sources/VoiceChat/OmlxClient.swift`:

```swift
struct OmlxConfig {
    var baseURL: URL = URL(string: "http://127.0.0.1:8000")!
    var chatModel: String = "mlx-community/Qwen3-Omni-30B-A3B-Instruct-8bit"
    var sttModel: String = "mlx-community/whisper-large-v3-mlx"
    var ttsModel: String = "prince-canuma/Kokoro-82M"
    var ttsVoice: String = "af_heart"
    var temperature: Double = 0.7
}
```

If omlx exposes the chat model under a short alias, you can use that instead of
the full HF repo id.

## Known gaps / TODO

- **Streaming chat** — currently non-streaming for simpler tool-call handling.
  Switch to SSE in `OmlxClient.chat` when latency becomes the bottleneck.
- **Direct audio-in to chat** — bypass the STT hop by sending the WAV as an
  `audio` content block in `/v1/chat/completions` once we verify omlx accepts
  Qwen3-Omni's multimodal input format end-to-end.
- **Streaming TTS playback** — `/v1/audio/speech?stream=true` returns chunked
  WAV; wire `AVAudioPlayerNode` to play as bytes arrive.
- **Sandboxing** — `run_shell` is unsandboxed by design. Tighten before
  pointing this at untrusted prompts.
